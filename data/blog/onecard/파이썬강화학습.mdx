# LLM 을 활용하게 된 계기

ChatGPT를 시작으로 LLM이 널리 알려지기 시작한 시기가, 제가 취업 준비를 하던 2022년 즈음이었던 것으로 기억합니다.
그 당시만 해도 주변 친구들 사이에서 “자기소개서를 LLM으로 작성하면 빠르고 퀄리티도 좋더라”는 이야기가 조금씩 들려오기 시작하던 때였습니다.

하지만 그 시절의 저는 LLM을 비롯한 AI 분야에는 큰 흥미가 없었습니다. 이유는 크게 세 가지였습니다.

> 1. 백엔드 개발과 아키텍처 공부에 집중해도 시간이 부족하다고 느낌
> 2. LLM이 생성하는 정보를 신뢰하기 어려움
> 3. LLM에 의존하면 스스로 성장하지 못할 것이라는 두려움

지금 돌이켜보면, 새로운 것을 시도하는 걸 좋아하는 제 성격상 사실 관심이 없었던 것은 아닙니다.
다만 “지금 공부하는 분야를 깊이 파야 한다”는 일종의 강박 때문에 의도적으로 눈을 돌리고 있었던 것 같습니다.

그러나 시간이 흐른 지금, LLM은 개발 전반에 걸쳐 활발히 활용되고 있으며 ‘바이브 코딩(Vibe Coding)’ 이라는 신조어까지 등장했습니다.
이제는 더 이상 외면할 수 없는 흐름이라 생각했고, 나 역시 적극적으로 활용해 보기로 마음먹었습니다.

LLM을 실제로 사용하면서 느낀 점은 두 가지였습니다.생활적인 측면에서는 새로운 분야를 배울 때의 진입 장벽을 낮춰준다는 점, 프로그래밍 측면에서는 무엇을 해야 하는지만 명확히 안다면 매우 강력한 도구가 될 수 있다는 점입니다.

이 두 가지 장점을 모두 활용할 수 있는 분야로 떠올린 것이 바로 강화학습(Reinforcement Learning) 이었습니다. 강화학습은 예전부터 언젠가 도전해 보고 싶던 분야였지만, 그동안은 교양 수준으로만 알고 있던 영역이었습니다. 이번 기회를 통해 LLM의 도움을 받아 본격적으로 강화학습을 구현해 보기로 결심했습니다.

# 원카드 게임을 선택한 이유

저는 개인적으로 카드 게임을 정말 좋아합니다. 유치원생 때 처음 해본 UNO와 원카드를 시작으로 하스스톤, 레전드 오브 룬테라, 매직 더 게더링 아레나 같은 디지털 카드 게임, 그리고 안드로이드: 넷러너, 아컴 호러 같은 보드게임까지 다양하게 즐겼습니다. 특히 하스스톤은 고등학교 2학년 때부터 대학교 3학년 때까지 꾸준히 플레이할 정도로 좋아했으며, 전설 랭크 달성과 투기장 12승도 경험했습니다.

이렇다 보니 언젠가 직접 카드 게임을 만들어보고 싶다는 개인적인 욕심이 있었습니다. 하지만 제대로 된 게임을 만들기 위해서는 프로그래밍뿐 아니라 기획력도 필요했고, 무엇보다 UI를 구현할 수 있는 게임 엔진 개발 역량(유니티·언리얼 등) 이 부족해 쉽게 도전하지 못했습니다.

그러던 중 회사에서 시스템 운영 지원용 UI를 개발하게 되었고 이때 공부를 위해 읽었던 「모던 리액트 Deep Dive」 책이 전환점이 되었습니다. 리액트로 컴포넌트를 다루다 보니, 굳이 게임 엔진이 아니더라도
카드 게임 정도는 직접 구현할 수 있겠다는 확신이 들었습니다.

게다가 원카드는 기획이 복잡하지 않고, 포커 카드 이미지 등 오픈소스 에셋이 이미 풍부하게 존재한다는 점도 매력적이었습니다. 이런 이유로 원카드 게임을 직접 만들어보기로 결정했습니다.

초기에 구현했던 원카드의 AI는 단순히 “낼 수 있는 카드 중 하나를 선택”하는 수준의 탐색 기반 로직이었습니다. 하지만 이번 프로젝트에서는 이 AI를 강화학습 기반으로 고도화해 보고자 했습니다.

# 잘못된 첫 설계

처음 게임을 개발할 때는 탐색 기반 AI + 싱글플레이어 + 멀티플레이어 기능 정도만을 목표로 삼았습니다. 그래서 싱글플레이어 모드는 별도의 서버를 거치지 않고, 모든 게임 로직을 클라이언트 단에서 처리하도록 설계했습니다.

하지만 이후 AI를 강화학습 기반으로 확장하려는 과정에서 이 구조가 예상보다 큰 제약으로 드러났습니다.
서버 환경 없이 클라이언트 내부에서 모든 상태와 액션이 처리되다 보니 AI의 학습에 필요한 상태, 보상, 정책 데이터를 안정적으로 수집하거나 시뮬레이션을 반복하기가 어려웠습니다.

이 문제를 해결하기 위해 게임 로직을 엔진 단위로 모듈화하고 Python 환경에서도 동일한 규칙을 사용할 수 있도록 브리지 서버 구조를 시도했습니다.
이를 통해 강화학습 환경을 구성하고 일부 실험은 가능했지만 엔진이 여전히 클라이언트에 종속되어 있다는 근본적인 한계는 남았습니다.

결국 AI 실험과 개발을 더 유연하게 진행하기 위해 클라이언트에 포함되어 있던 싱글플레이어 게임 로직을 백엔드 API로 완전히 분리할 예정입니다.

아래의 내용은 이 결론에 닿기까지 겪은 시행착오에 대한 내용입니다.

# ONE CARD AI & 게임 엔진 리팩터링 여정

처음엔 AI 쪽 지식이 전무한 채로 “게임에 인공지능을 붙여보자”는 막연한 욕심만 있었습니다. 다행히도 LLM에게 “이럴 땐 어떻게 하지?”라고 끊임없이 물어볼 수 있었고, 대답을 따라가다 보니 어느새 RL 환경과 브리지 서버까지 만들고 있더군요. 마치 든든한 페어 프로그래머를 곁에 둔 느낌이었고, 그 덕분에 겁먹지 않고 새로운 영역에 뛰어들 수 있었습니다.

## 1. 시작: 클라이언트 중심 엔진에서의 고민

원래 ONE CARD 웹게임은 Next.js 클라이언트에서 모든 게임 로직을 처리하고 있었습니다. 싱글 플레이어 모드라면 이 방식도 충분했지만, 멀티플레이 계획과 강화학습 실험을 생각하면 몇 가지 문제가 드러났습니다.

- **권위(authoritative) 상태가 클라이언트에 있다** → 치트 방지, 멀티플레이 동기화, 로그 수집이 어렵다.
- **AI 행동이 클라이언트에 묶여 있다** → 다른 환경(서버, Python)에서 재사용하기 힘들다.
- **강화학습 환경으로 변환하기 까다롭다** → 상태 전이가 React 훅과 얽혀 있어 재현성 있는 시뮬레이션을 만들기 어렵다.

이 문제들을 해결하기 위해 엔진을 모듈화하고, 이후 RL과 서버 브리지를 붙여 나가는 여정을 시작했습니다.

## 2. 엔진 모듈화: gameEngine.ts 도입

첫 단계는 기존 `useOneCardGame` 훅에서 reducer/상태 전환 로직을 분리하는 일이었습니다. `lib/engine/gameEngine.ts`는 순수 함수 형태로 상태 전이를 담당하고, 훅은 이 엔진을 감싸는 형태로 단순화됩니다. 주요 포인트는 다음과 같습니다.

- `createInitialState`, `step`, `applyActions` 함수로 상태 전이를 순수 함수화.
- React나 브라우저 의존성을 제거해 서버/브리지/Python 환경에서도 동일한 로직 사용 가능.
- 추후 강화학습 환경이 동일한 엔진을 호출할 수 있도록 기반 마련.

이 단계 덕분에 클라이언트 외부에서도 동일한 규칙을 재사용할 수 있는 토대가 생겼습니다.

### 구현 상세

- `GameEngineContext` 구조체로 덱 생성기, 난이도 규칙, 보상 전략 등을 주입할 수 있게 만들어 테스트와 실험이 수월합니다.
- 상태 전이는 `reduceTurn(state, action)` 한 함수에 모았고, 각 규칙(역순, 대미지 누적, 조커 처리 등)은 작은 pure helper로 쪼개 side effect 없이 동작합니다.
- `serializeState`/`deserializeState`를 추가해 브라우저 저장소, 서버, RL 브리지 간에 동일한 JSON 포맷을 주고받을 수 있도록 했습니다.
- 훅(`useOneCardGame`)은 이제 엔진에 액션만 전달하고, UI 이벤트/드래그 앤 드롭 같은 뷰 로직만 담당하도록 역할이 명확히 분리됐습니다.

## 3. 강화학습을 위한 Python 인프라: engine-bridge & python-rl

엔진을 모듈화한 뒤, Python에서 시뮬레이션을 돌릴 수 있는 구조를 만들었습니다. 본격적인 구현 설명에 앞서, AI 초심자도 흐름을 이해할 수 있도록 강화학습 기본 개념을 짚고 넘어갑니다.

### 강화학습 한눈에 보기

- **에이전트(Agent)**: 결정을 내리는 주체입니다. ONE CARD에서는 플레이어 0이 됩니다.
- **환경(Environment)**: 에이전트를 둘러싼 세계이며, 우리는 게임 엔진을 그대로 환경으로 사용합니다.
- **상태(State)**: 에이전트가 현재 상황을 파악하기 위해 받는 정보입니다. 손패, 공개 카드, 피해량 등이 포함됩니다.
- **행동(Action)**: 에이전트가 취할 수 있는 선택지입니다. 카드 내기 또는 드로우가 대표적입니다.
- **보상(Reward)**: 행동 결과에 따라 에이전트가 받는 점수입니다. 승리하면 +1, 불법 행동이면 -1처럼 설계합니다.

이 흐름이 반복되며 에이전트는 좋은 보상을 주는 전략을 학습합니다. 학습 과정에서 가장 많이 쓰인 알고리즘 중 하나가 **PPO(Proximal Policy Optimization)**입니다.

#### PPO 간단 설명

PPO는 “정책(Policy)”이라 부르는 확률 분포를 조금씩 업데이트하며 안정적으로 학습하는 방식입니다. 과거 정책과 새 정책이 너무 멀어지지 않도록 제약(clip)을 걸어, 초심자도 비교적 쉽게 튜닝할 수 있는 장점이 있습니다. Stable-Baselines3 라이브러리가 PPO를 고수준 API로 제공하므로, 우리는 Hyperparameter만 CLI 인자로 넘겨도 학습을 수행할 수 있습니다.

간단한 반복 구조는 아래처럼 표현할 수 있습니다.

```python
env = OneCardEnv(endpoint="http://localhost:4000")
model = PPO("MlpPolicy", env, learning_rate=3e-4)
model.learn(total_timesteps=200_000)
```

`OneCardEnv`가 게임 상태를 Box 벡터로 바꿔 주고, `PPO`는 이 벡터를 받아 어떤 행동을 할지 결정합니다. 학습이 끝나면 `.zip` 파일로 정책이 저장되고, 추론 시 다시 불러 사용합니다.

### 3.1 engine-bridge
- Node.js Express 서버(`engine-bridge/src/engine-server.js`)를 만들고, `onecard-client/dist/engine/gameEngine.js`를 require하여 REST API(`/reset`, `/state`, `/step`)를 제공합니다.
- 이렇게 하면 Python이나 다른 서비스가 HTTP로 게임 엔진을 호출할 수 있습니다.

#### 구현 상세

- `tsup`으로 번들한 CommonJS 엔진을 로드하고, 요청마다 `GameEngine` 인스턴스를 생성해 순수 상태를 관리합니다.
- `/reset`은 전달받은 설정을 `createInitialState`에 넘겨 새 상태를 만든 뒤 메모리 캐시에 저장하고 JSON으로 반환합니다.
- `/step`은 payload를 바로 `engine.step`에 연결하고, 에러가 나면 HTTP 400으로 메시지를 내려 Python 쪽에서 학습을 중단하도록 했습니다.
- 과정 중 생기는 `@/lib` 경로 문제는 `module-alias` 대신 `tsconfig.paths.json`을 그대로 사용하도록 `NODE_PATH`를 설정해 해결했습니다.

### 3.2 Gym 환경(`python-rl/gym_env.py`)
- Gymnasium 인터페이스를 따르는 `OneCardEnv`를 작성했습니다.
- 관측 벡터 구조(`ObservationSpec`, `ObservationEncoder`)와 행동 디코딩 로직을 정리하고, 현재 플레이어 시점만 노출되도록 `_resolve_to_agent_turn`에서 AI 턴을 자동으로 소모합니다.
- 보상 함수(`default_reward`)를 분리해 추후 커스터마이즈 가능하게 했습니다.

#### 구현 상세

- `ObservationSpec` dataclass가 벡터 길이를 계산하고, `ObservationEncoder`는 손패/버린 카드/대미지를 원-핫과 정규화 값으로 변환합니다.
- `_decode_action`은 Gym의 정수 액션을 실제 게임 액션(카드 인덱스, 드로우 여부)로 바꾸며, numpy dtype 문제를 막기 위해 모든 값을 `int()`로 캐스팅합니다.
- `_resolve_to_agent_turn`은 서버가 돌려준 상태에서 `currentPlayer`가 에이전트가 될 때까지 자동으로 `NEXT_TURN`을 호출해, AI가 연속으로 여러 장을 내려도 한 번의 관측만 전달됩니다.
- `default_reward`는 승리(+1), 패배(-1), 손패 변화량/대미지 해소 같은 shaping 항목을 조합해 PPO 학습이 안정적으로 수렴하도록 돕습니다.

핵심 메서드의 흐름은 다음 코드처럼 단순한 while 루프로 표현됩니다.

```python
def _resolve_to_agent_turn(self, state):
    buffer = state
    while buffer["currentPlayer"] != self.agent_index:
        buffer = self._post_json("/step", {"action": {"type": "NEXT_TURN"}})["state"]
    return buffer
```

이렇게 해서 환경이 “플레이어 차례만” 관찰하게 만들고, 초보자도 턴 동기화 문제에 신경 쓰지 않고 학습 코드를 작성할 수 있습니다.

### 3.3 학습/추론 스크립트
- `train.py`: Stable-Baselines3 PPO로 학습. `DummyVecEnv`, 체크포인트 콜백 등을 사용해 CLI 인자 기반으로 동작합니다.
- `inference.py`: 학습된 모델을 불러와 엔진 브리지와 상호작용하며 플레이를 시뮬레이션합니다.
- 학습 중 발생한 JSON 직렬화 오류(numpy.int64)는 `_decode_action`에서 `int()` 캐스팅으로 해결했습니다.

#### 구현 상세

- `train.py`는 `argparse`로 엔드포인트, 시드, timesteps, 체크포인트 디렉터리를 받고, `EvalCallback`과 `CheckpointCallback`을 동시에 설정해 학습 중간 결과를 기록합니다.
- 벡터라이즈드 환경으로 `DummyVecEnv`를 쓰지만, 멀티 프로세스 확장을 염두에 두고 `make_env` 팩토리 함수를 별도로 두었습니다.
- `inference.py`는 학습 결과 `.zip`을 불러오고, step마다 `OneCardEnv`의 상태를 사람이 읽기 쉬운 문자열로 변환해 로그를 남깁니다.
- 두 스크립트 모두 브리지 서버 가용성을 먼저 확인하고, 연결 실패 시 명확한 에러를 출력해 디버그 시간을 줄였습니다.

초심자에게 익숙한 CLI 예시는 다음과 같습니다.

```bash
# 학습
python train.py --endpoint http://localhost:4000 --timesteps 300000 --seed 42

# 추론
python inference.py --endpoint http://localhost:4000 --model checkpoints/best_model.zip
```

`train.py`가 체크포인트를 자동 저장하므로, 추후에는 바로 `inference.py`로 모델을 불러 플레이 로그를 확인하면 됩니다.

## 4. 환경 동작 개선: AI 턴 자동 처리

초기 버전의 환경은 플레이어 0의 행동 직후 상태만 반환했기 때문에, 추론 시 상대 AI가 아무것도 하지 않는 것처럼 보였습니다. `_resolve_to_agent_turn` 함수를 도입해 다음과 같이 개선했습니다.

1. `reset()`이나 `step()` 후 반환된 상태에서 `currentPlayer`가 AI면 자동으로 `NEXT_TURN` 액션을 반복 호출.
2. 에이전트가 다시 행동해야 하는 상태가 될 때까지 버퍼링한 뒤 관측을 반환.
3. 보상 계산은 이전 상태와 “AI 턴 처리가 끝난 상태”를 비교해 수행.

이제 학습/추론 모두 “사용자 턴 → 상대 AI 자동 진행 → 다시 사용자 턴” 흐름으로 자연스럽게 진행됩니다.

## 5. 교훈 및 다음 단계

- **엔진 재사용성**: 규칙을 순수 함수로 모듈화하니 클라이언트/서버/Python에서 모두 같은 로직을 재사용할 수 있었습니다.
- **브리지 설계**: 간단한 Express 서버만 있어도 다른 언어(예: Python)에서 쉽게 연결할 수 있습니다.
- **Gym 환경 설계 중요성**: 관측/보상/행동 공간을 명확히 정의하고, AI 턴을 자동으로 처리하니 RL 학습이 훨씬 안정적입니다.
- **Docstring & 타입**: Google 스타일 Docstring을 적용하면서 타입 구조와 의도를 명확히 할 수 있었고, Pylance 경고도 줄였습니다.

### 다음 단계
1. **서버 권위 구조 도입**: 실제 서비스에서도 엔진을 서버로 이전해 멀티플레이/로그 수집을 준비.
2. **AI 전략 확장**: 강화학습 모델을 서버 측 추론으로 연결하거나, 난이도별 전략을 다양화.
3. **시드/재현성**: 엔진 브리지에 시드를 전달해 완전한 재현성을 확보.
4. **멀티플레이 모드**: 현재 단일 플레이어 환경을 확장해 여러 에이전트/플레이어가 동시에 접속 가능한 구조로 발전.
5. **백엔드 우선 구축 후 RL 재도전**: 게임 로직을 완전히 서버 권위 형태로 옮긴 뒤 다시 강화학습을 돌릴 계획입니다. 이렇게 하면
   - 동일한 REST/gRPC API를 학습·추론 양쪽에서 그대로 쓰므로, 브리지나 의존성 문제로 실험이 중단되지 않습니다.
   - 멀티플레이/로그/치트 방지 등 실 서비스 요구사항과 RL 실험 환경을 동시에 만족할 수 있습니다.
   - 서버에서 상태 전이를 일관되게 기록할 수 있어 데이터 세트 구축과 리플레이 기반 학습(offline RL)로도 확장할 수 있습니다.
   결국 “먼저 백엔드 구축 → 권위 상태 확보 → 그 위에서 RL 파이프라인 재구성” 순서를 따르면, 현재의 임시 브리지보다 훨씬 안정적으로 학습을 반복할 수 있을 것입니다.

이 과정에서 비록 엔진이 아직 클라이언트에 남아 있긴 하지만, `engine-bridge`와 `python-rl`을 통해 이미 “외부 서비스에서 같은 규칙을 사용한다”는 흐름을 마련했습니다. 다음 단계로 서버 권위 구조와 멀티플레이 지원을 붙이면, ONE CARD 프로젝트가 하나의 게임 AI 플랫폼으로 확장되는 기반이 마련될 것입니다.

마지막으로, 이번 여정에서 얻은 가장 큰 교훈은 **“한 번 만든 규칙을 어디서든 재사용할 수 있도록 만드는 것”**이 장기적으로 가장 큰 생산성을 준다는 점입니다. 엔진 → 브리지 → Gym 환경 → 학습/추론 파이프라인이 유기적으로 연결되면서, 이제는 어떤 프로토타입을 만들더라도 동일한 규칙과 데이터를 공유할 수 있게 되었습니다. 이 일관성이 앞으로의 기능 추가와 연구 실험 속도를 더욱 끌어올릴 것입니다.
